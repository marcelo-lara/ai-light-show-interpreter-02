# ai-light-show-interpreter-02: Stage Virtual Canvas Interpreter

## 1. Overview
The `ai-light-show-interpreter-02` is the core "rendering engine" of the AI Light Show ecosystem. It functions as a **rasterizer and baker** that transforms mathematical intent and audio-reactive features into a production-ready, sample-accurate DMX binary stream.

By utilizing a **Stage Virtual Canvas** approach, this module moves away from traditional "fixture-based" chasers. Instead, it treats the lighting rig as a coordinate-aware 2D mesh, evaluating mathematical equations (pulses, waves, and gradients) to determine the state of every light.

## 2. Technical Specifications

*(For detailed JSON schemas and pipeline architecture, please refer to the [`docs/`](docs/) directory).*

### Input: Semantic Intelligence Artifacts
The interpreter consumes artifacts generated by the song analyzer:
* **Song Structure (`sections.json`):** Used to determine section-based rendering logic by mapping timestamps to section labels (e.g., applying different mathematical patterns for verses vs. choruses or ambient vs. drop sections).
* **Beat and Duration Metadata (`data/artifacts/<Song - Artist>/essentia/beats.json`):** Provides the authoritative duration and beat grid used for render timing and cue validation.
* **Audio Frequency Data (`data/artifacts/<Song - Artist>/essentia/fft_bands.json`):** Provides raw, frame-by-frame energy levels for exactly 5 frequency bands. The interpreter dynamically applies temporal smoothing (attack/release envelopes) to these values during rendering to prevent visual jitter.
* **Event Timeline (`data/artifacts/<Song - Artist>/lighting_events.json` [Optional]):** Triggers instantaneous mathematical "Hard Cuts" or coordinate shifts in the 2D mesh based on recognized audio events. If this file is missing or invalid, the interpreter falls back to dynamically detecting transients (sudden energy spikes) directly from the **Audio Frequency Data** stream to trigger impacts.

### The Bridge: Stage Virtual Canvas
A hybrid configuration model defines the virtual-to-physical mapping:
* **Canonical Rig Inputs:** `data/fixtures/fixtures.json` and `data/fixtures/pois.json` define fixture identity, physical placement, and named moving-head targets.
* **Virtual Canvas Metadata:** `src/config/stage_virtual_canvas.json` defines app-specific sampling/grouping metadata layered on top of the rig inputs.

### Output: Production-Grade DMX Binary
**Strict Output Rule:** This repository ONLY writes compiled show files to the `data/shows/` directory. All other files and directories within the `/data` folder are strictly **READ-ONLY** for this repository. No other artifacts or intermediate data are generated.

The final output is a show file following the **DMX Binary Specification** (e.g., `data/shows/Cinderella - Ella Lee.main-show.dmx`):
* **Header:** Metadata including file version, frame rate (e.g., 50 FPS), and total universe count.
* **Payload:** A flat, seekable byte-array buffer. Each frame contains raw DMX values (512 bytes per universe).
* **Performance:** Designed for high-reliability playback with zero computational overhead during the performance.

## 3. Rendering Pipeline (The Baker)

1.  **Coordinate Lookup:** For each fixture defined in the **Stage Virtual Canvas**, the system retrieves its $(x, y)$ position.
2.  **Smoothing & Evaluation:** For every 20ms frame (at 50 FPS), the system applies configurable exponential smoothing to the raw audio bands, then evaluates the current mathematical "state" of the canvas driven by these reactive multipliers.
3.  **Rasterization:** The system "samples" the color and brightness at each fixture's coordinate.
4.  **Baking:** Sampled results are written directly into the binary DMX frame.

## 4. Running the Interpreter in Docker

1. Build the Docker image:

```bash
docker compose build
```

2. Start the default browser-driven UI stack:

```bash
docker compose up -d --build
```

3. Open the Canvas Output UI:

```text
http://localhost:3300
```

4. Pick a song from the header dropdown. The UI sends a `select_song` message over the WebSocket on port `3301`, the backend bakes the `.dmx` file, and the Play button becomes available when the bake completes.

5. If you want the original terminal selector instead of the browser flow, run the CLI directly:

```bash
docker compose run --rm light-show-cli --show-name main-show
```

The selector will mark songs that are missing analyzer artifacts with ` [MISSING ARTIFACT]`.

6. If you need to render non-interactively, you can still pass `--song` and `--show-name` and the tool will write the compiled DMX to `data/shows/`.

```bash
docker compose run --rm light-show-cli --song "Cinderella - Ella Lee" --show-name main-show
```

7. To export a debug SVG of the current fixture and POI layout without opening the player:

```bash
docker compose run --rm light-show-cli --export-layout
```

The SVG is written to `./stage-layout.svg` by default and includes moving-head beam lines pointing to each fixture's current default `target_poi`.

8. Verify the output file exists:

```bash
ls -lh data/shows/*.dmx
```

> Note: `./data` is mounted read-only, while `./data/shows` is mounted writable. All other input data remains protected.

## 5. Professional-Grade Features
* **Separation of Concerns:** Intent is stored in human-readable math/coordinates; execution is stored in lean binary.
* **Sample-Accurate Sync:** Pre-rendering eliminates real-time jitter and CPU-induced lag.
* **Volumetric Scaling:** Upgrading a rig only requires updating the **Stage Virtual Canvas** coordinates; the artistic logic remains untouched.
